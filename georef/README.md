# georef — Per-Project Coordinate Authority

Every LATTICE project has **one** `georef.config.json` at its project root. That file is the master coordinate authority: every coordinate stored anywhere in Pixeltable derives from it. It seeds `lattice/bridge/project_georef`. Survey data wins over IFC; IFC wins over GPS; GPS wins over Google Maps; Google Maps wins over OSM. The full priority order is the `source_priority` array in the config.

## How a project gets georeferenced

```
project root/
└── georef.config.json   ←─ master file, hand-edited or generated by an ingest script

         │  POST /v1/georef/ingest/config  (sidecar)
         ▼
lattice/bridge/project_georef   ←─ canonical row, 67 columns

         │  POST /v1/georef/compute-transforms/{project_id}
         ▼
4x4 transform matrices written back into the same row:
  transform_vw_to_wgs84
  transform_wgs84_to_ecef
  transform_project_to_utm
  transform_ifc_to_wgs84
```

After that, every downstream component reads from `project_georef`:

- **VW bridge** reads `vw_origin_*` + `vw_rotation_deg` to align VW internal coords to WGS84
- **Cesium globe** reads `boundary_geojson` for the site outline
- **deck.gl** reads `bounding_box_json` to set its initial viewport
- **ThatOpen viewer** reads `transform_ifc_to_wgs84` to drape the IFC over real-world terrain
- **Point cloud / splat ingest** reads `epsg_code` to reproject input data to the project CRS

## Cardinal rule

Never store raw VW internal coordinates in `lattice/bridge/*` tables. Always normalize through `project_georef` first. The `lattice/bridge/ifc/ifc_elements.geom_point_wkt` column carries WGS84 `POINT(lon lat)` — nothing else.
