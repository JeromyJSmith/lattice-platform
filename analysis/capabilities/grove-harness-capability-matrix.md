## Notes on decision policy

The variant defaults applied above are correct for the following reasons:

1. **Architecture rows (grove-sfa-pattern, grove-l2-wrapper-pattern, grove-variant-taxonomy, grove-tool-manifest-format, grove-marker-contract, grove-dispatch-helper, grove-duckdb-cache-substrate)** are marked `candidate` because they define the substrate contract for all SFAs. These are primarily documentation + template validation, low risk, and immediately useful for understanding Grove's design. The dispatch-helper is a small production utility already used in the existing codebase.

2. **H-I Host Ingest SFAs not bound to VW bridge** (init_core_pixeltable_schema, init_ledger, init_caches, setup_grove, build_grove_audit_manifest, ingest_database, ingest_audit_manifest, ingest_ifc, ingest_dxf, build_external_tool_indexes) are marked `candidate` and assigned to `meta-harness` or `georef-harness` because they are pure host-side infrastructure with no dependency on the VW bridge. They are low-risk read-only ingest and setup SFAs. Candidates for immediate first proof runs.

3. **H-D Host Derive/Diagnostic SFAs** (refresh_harness_cache, build_qto_views, build_visual_qa_views, build_geojson_export, build_meta_harness_views, cost_views, qto_reconcile, reconcile_la_composition, budget_validate, index_external_tool_repos, promote_external_tool_skills, derive_scene_plant_locations, derive_scene_plant_locations_from_dxf, build_plant_tool_batch_plan, record_plant_sync_tables, qa_plant_exactness, derive_plant_tool_exact_payload) are read-only derivations over Pixeltable and DuckDB marts. Those without downstream dependencies (reconcile_la_composition, refresh_harness_cache, etc.) are marked `candidate`. Those with pending wrapper completion (derive_plant_tool_exact_payload, qa_plant_exactness, build_plant_tool_batch_plan, record_plant_sync_tables) are marked `defer` with reason `awaiting-upstream-dep`.

4. **H-O Host Output/Orchestrate SFAs** (build_project_report, build_tool_catalog, drop_zone_daemon) are marked `candidate`. The drop_zone_daemon is a long-running daemon variant and flagged as such in notes; its verification target is the status JSON endpoint rather than a done marker.

5. **P-R Polymorphic Read** (extract_plant_positions, extract_landscape_area_composition) and **P-D Polymorphic Diagnostic** SFAs are marked `defer` with reason `awaiting-upstream-dep` because vwx-mcp transport is not yet wired. Grove used an AppleScript bridge; LATTICE uses vwx-mcp. Once vwx-mcp integration is complete, these rows transition to `candidate`.

6. **P-W Polymorphic Write** SFAs (apply_document_georef, bootstrap_plant_3d_pins, place_survey_stakes, etc.) are marked `defer` with reason `awaiting-upstream-dep` for the same vwx-mcp integration reason, plus LATTICE's write-gating policy. Recover_vw_origin_georef is marked `high` value due to its combined recovery scope; probe_file_health_check is marked `low` value and deferred to Phase 3 as experimental.

7. **Cost-engine SFAs** (init_cost_schema, budget_ingest, budget_validate, cost_views, build_qto_views, qto_reconcile, ingest_plant_supplier_specs) are marked `defer` with reason `out-of-scope-for-current-phase` and assigned to `ddc-harness`. DDC integration is Phase 4 of LATTICE. These are high-value rows but not in scope for Wave 1.

8. **B-R Browser Read SFAs** (lookup_ifc_class, scrape_plant_supplier) are marked `defer` with reason `awaiting-upstream-dep` because browser-harness integration is not yet wired. Once browser-harness is connected, these move to `candidate`. lookup_ifc_class is assigned to `genai-harness` (knowledge lookup); scrape_plant_supplier is assigned to `genai-harness` (retrieval + ingest pipeline) but could also be assigned to `vw-itwin-harness` once supplier data flows are defined.

9. **Marimo notebooks** (grove-marimo-control-plane, grove-marimo-cost-engine) are marked `defer` with reason `out-of-scope-for-current-phase`. LATTICE Wave 1 uses TanStack Start for the UI. Marimo is a Phase 2+ exploration target for operator control surfaces and Phase 4 for cost-engine rendering.

10. **VW-ingest wrappers** (ingest_vw_extract, ingest_vw_safe_extract, marpa_programmatic_place_true_plant) are marked `defer` with reason `awaiting-upstream-dep` because VW bridge data flow is not yet complete. Once vwx-mcp is wired and wrapper completion is done, these move to `candidate`.

The transition from `defer` to `candidate` happens when:
- **vwx-mcp is wired** (Phase 2 VW Bridge): all P-* and VW-bridge-dependent H-I rows unblock
- **browser-harness is integrated** (Phase 2): B-R rows unblock
- **DDC integration is complete** (Phase 4): cost-engine and supplier-data rows unblock
- **wrapper completion** (Phase 2): unwrapped SFAs and daemon variants unblock
